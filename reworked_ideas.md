# Health Planner — Концепція проєкту

> **Статус:** Чернетка ідей (pre-requirements)
> **Домен:** Healthcare / Personal Health Management
> **Платформи:** Mobile-first (iOS, Android), потенційно Web
> **Останнє оновлення:** 2026-05-06

---

## 1. Бачення продукту

### 1.1 Основна ідея
Мобільний застосунок для **персонального контролю здоров'я**: планування, збереження та інтелектуальна обробка медичних аналізів і обстежень. Допомагає користувачу та його родині не пропускати чекапи, розуміти результати аналізів і готуватися до візитів до лікаря.

### 1.2 Цільова аудиторія
- Люди, які свідомо стежать за своїм здоров'ям (preventive health).
- Батьки, які ведуть медичну історію дітей.
- Опікуни немічних/літніх родичів.
- Користувачі в ЄС та Україні (на старті).

### 1.3 Ключові цінності для користувача
1. **Єдине місце** для всієї медичної історії родини.
2. **Розуміння** результатів аналізів простою мовою (AI-пояснення).
3. **Проактивність** — нагадування й планування з урахуванням реальних термінів очікування в країні.
4. **Контекст** — підсвічування неочевидних відхилень і комбінацій показників.

## 2. Функціональний скоуп

Усі функції згруповані за пріоритетом для майбутнього MVP/roadmap-планування.

### 2.1 MVP (Core)

#### F1. Контроль запланованих аналізів та обстежень
- Дашборд з найближчими активними подіями (що, коли, дедлайн запису).
- Часові вікна для запису з урахуванням **локальних термінів очікування** (напр., МРТ у Польщі — до 6 місяців).
- Push-нагадування: «Час записатись на УЗІ живота».
- План на найближчий місяць + довгострокові події (за пів року).

#### F2. Каталог бажаних аналізів (моніторинг показників)
- Бібліотека базових аналізів + можливість пошуку та додавання власних.
- По кожному аналізу — список релевантних показників.
- **Норми двох рівнів:**
  - *Лабораторні* (формальні референсні діапазони).
  - *Клінічні/реальні* (наприклад, феритин у нижній межі лабораторної норми вже є проблемою — підсвічуємо).
- Підтримка **крос-показникових правил** (напр., глюкоза висока + інсулін у нормі = підозра на інсулінорезистентність).
- Користувач може кастомізувати норми під себе (за рекомендацією лікаря).

#### F3. Парсинг та зберігання аналізів
- Завантаження файлу: **PDF, зображення (JPG/PNG), фото камерою**.
- AI-екстракція показників (легка LLM-модель, напр. Gemma / on-device або сервер).
- Користувач **валідує** розпізнаний текст перед збереженням.
- **Колірне маркування показників:**
  - 🟢 Зелений — у нормі.
  - 🟡 Жовтий — погранично.
  - 🔴 Червоний — поза нормою.
  - 🚨 Критично — нотифікація з номером локальної швидкої.
- Зберігаємо **оригінальний файл** + розпарсені дані + AI-резюме простою мовою.
- Після збереження дашборд показників автоматично оновлюється.

#### F4. Багатомовність
- Старт: **🇬🇧 English, 🇩🇪 Deutsch, 🇺🇦 Українська**.
- Архітектура з самого початку має дозволяти **легке додавання мов** (i18n-ready).

#### F5. Автентифікація
- Логін через **Google, Apple, Meta (Facebook)**.
- Email/password — опційно.
- Облік юридичних вимог (див. §4).

#### F6. Профіль та стартовий онбординг
- Опитувальник для збору базових даних (стать, вік, хронічні стани, алергії, локація/країна).
- Дані використовуються для персоналізації рекомендацій по аналізах.

### 2.2 Post-MVP (важливо, але не на старті)

#### F7. Підготовка до візиту до лікаря
- Швидка вибірка профільних показників за період.
- Експорт у вигляді PDF / shareable view з **динамікою** (графіки змін).
- Лікар одразу бачить тренди (наприклад, ріст глюкози).

#### F8. Опікунські акаунти (multi-profile management)
- Один користувач керує **кількома профілями** (дружина → акаунт чоловіка + акаунт дитини).
- Ролі: власник, опікун, переглядач.
- Юридична модель згоди (consent) — критично, див. §4.

## 4. Юридичні та комплаєнс-питання ⚖️

> **Критично:** медичні дані = особлива категорія персональних даних. Помилки тут = штрафи й репутаційні ризики.

### 4.1 Регуляторні рамки до вивчення
- **GDPR** (ЄС) — Art. 9 (special categories), DPIA обов'язкова.
- **Закон України «Про захист персональних даних»** + медична таємниця.
- **HIPAA** — якщо є плани на ринок США.
- **MDR (Medical Device Regulation)** — чи вважатиметься застосунок medical device? Залежить від функцій (особливо AI-інтерпретація). Потрібна юридична консультація.

### 4.2 Ключові питання для дослідження
1. Зберігання файлів аналізів — де (юрисдикція серверів), скільки, шифрування at-rest/in-transit.
2. Згода користувача — granular consent, право на видалення, експорт даних.
3. Опікунські акаунти — як легально оформити доступ одного користувача до даних іншого (включно з дітьми та недієздатними).
4. AI-інтерпретація — disclaimer'и, відповідальність за помилкові рекомендації.
5. Cross-border data transfer (Україна ↔ ЄС).

### 4.3 Базові вимоги до архітектури
- Шифрування медичних даних на клієнті та сервері.
- Audit log усіх доступів до медичних записів.
- Можливість повного експорту й видалення акаунта.
- Мінімізація даних (зберігати лише необхідне).

## 5. Технологічні рішення

### 5.1 Стек (попередній)
| Шар                | Вибір                                                      | Обґрунтування                                       |
| ------------------ | ---------------------------------------------------------- | --------------------------------------------------- |
| **Mobile**         | Flutter + Dart                                             | Один кодбейс на iOS/Android, потенційно web/desktop |
| **Backend**        | TBD (.NET / Go / Node)                                     | Залежить від команди                                |
| **AI/OCR**         | Gemma / локальна LLM + OCR (Tesseract, ML Kit)             | Privacy-first, бажано on-device де можливо          |
| **БД**             | PostgreSQL + шифрування полів                              | Перевірений стек для медданих                       |
| **Сховище файлів** | S3-сумісне з encryption + KMS                              | З урахуванням юрисдикції                            |
| **Auth**           | OAuth 2.0 (Google/Apple/Meta) + Keycloak або Firebase Auth | —                                                   |

### 5.2 Локалізація даних
**Питання для опрацювання:** норми та назви аналізів відрізняються між країнами.
- Підхід: **спільний канонічний каталог аналізів** (LOINC-коди?) + локалізовані назви + локалізовані норми (країна, стать, вік).
- Архітектурний наслідок: схема БД має враховувати `analysis_type → localized_name`, `reference_range → (country, sex, age_range)`.

### 5.3 AI-компонент
- **Парсинг:** OCR → структурований JSON показників.
- **Інтерпретація:** LLM генерує пояснення простою мовою + детектує критичні комбінації.
- **Валідація користувачем** обов'язкова перед збереженням (юридичний buffer).

## 6. Бізнес-модель та монетизація 💰

### 6.1 Мета
Безкоштовний застосунок з широким охопленням → монетизація без шкоди для UX.

### 6.2 Гіпотези монетизації (потребують дослідження)
1. **Freemium**: безкоштовне ядро + Premium (необмежене сховище аналізів, AI-інтерпретація, кілька профілів родини, розширена аналітика).
2. **Family plan** — підписка для опікунських функцій.
3. **B2B2C через клініки/лабораторії** — інтеграції з лабораторіями (партнерські відрахування).
4. **Donation / pay-what-you-want** — як етичний варіант.
5. ❌ **Реклама** — не підходить для медичної ніші (довіра + юридика).
6. ❌ **Продаж знеособлених даних** — токсично репутаційно, навіть якщо легально.

### 6.3 Ліцензування / IP
- Кодова база — **закрите комерційне ПЗ** (proprietary).
- Можливі open-core частини (наприклад, каталог аналізів).
- Торгова марка — реєстрація у відповідних юрисдикціях.
